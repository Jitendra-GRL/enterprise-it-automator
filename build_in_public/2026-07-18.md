# Build in Public — 2026-07-18: The gaps between "portfolio project" and "enterprise-grade"

I spent this session doing something I don't usually do: instead of adding another
feature to the [MCP-Enabled Enterprise IT Automator](https://github.com/JitendraPrabhu-l/enterprise-it-automator),
I audited it against what production MCP/agent systems actually need in 2026 —
pulling from OWASP's MCP security cheat sheet, the Cloud Security Alliance's
agentic-MCP guidance, NIST's agent red-teaming research, and how enterprise
buyers actually evaluate this stuff (SOC2, FinOps, SCIM).

Posting the honest gap list. Not the highlight reel.

---

## Update, same day: all 7 Tier-1 gaps closed

Went back through the list below and resolved every Tier-1 item:

1. **MCP OAuth 2.1 + token exchange** — `MCP_SERVER_TOKEN` is now the admin
   credential only; `POST /token/exchange` mints short-lived, domain-scoped
   JWTs, and a `tools/call` outside a scoped token's domains gets `403
   insufficient_scope`. Self-issued, not a full external IdP — right-sized,
   same as everything else here.
2. **Tool-description integrity check** — every discovered tool is hashed
   against a committed baseline; drift is logged+metriced by default, or a
   hard `FAILED` ticket in strict mode. CI fails if the baseline goes stale.
3. **Tamper-evident audit log** — a hash chain over every `AuditLog` row,
   verifiable on demand via `GET /audit/verify`. Explicitly not WORM
   storage; the honest caveat is in the code, not just this post.
4. **SBOM** — turned out this was already shipping. Corrected the record
   below instead of quietly deleting the wrong claim.
5. **Adversarial fuzzing** — a 10-case live-model prompt-injection corpus,
   run weekly (and on-demand) against the real configured LLM, not
   recorded fixtures.
6. **Backup/DR isolation** — automated backup script + Helm CronJob using
   a credential the app itself never holds, plus a restore-drill script
   that actually proves a dump is recoverable, not just that a file exists.
7. **Org-level cost governance** — per-client and org-wide daily token
   budgets, enforced at submission and mid-run, closing a gap the code's
   own docstring had flagged as "too large a change" a few weeks ago.

Same test suite, same coverage floor, same review bar the rest of the
project already holds itself to. Details on request — the diff is bigger
than one post.

## What's already real (so the gap list below isn't "starting from zero")

- **Server-side approval gate**: sensitive MCP tool calls (disable_user, revoke_access)
  require a pre-verified, single-use approval token checked against the *exact*
  tool + arguments. This is enforced inside the MCP server itself — not a prompt
  telling the LLM to ask nicely. An LLM can't talk its way past it.
- **Eval-gated guardrails**: six golden tickets (including a prompt-injection
  refusal case) replayed through the real LangGraph graph in CI, plus a
  separate live-model eval run. This setup already caught a real bug — a stale
  `.env` value silently exempting `enable_user` from human approval — and
  drove a measured model swap after scoring the 8B default at 3/6 vs. 5/6 for
  a larger model.
- **Observability**: OpenTelemetry spans using `gen_ai.*` semantic conventions
  (the actual current standard, not a custom schema), Prometheus RED +
  domain metrics, Grafana + Alertmanager with seven runbook-linked alert rules.
- **Reliability primitives**: idempotency keys, retry/backoff with jittered
  backoff, circuit breakers per MCP backend, a Postgres advisory lock so SLA
  sweeps don't double-fire across replicas.
- **Supply chain**: mypy strict, pip-audit, Trivy image scanning, Dependabot,
  Alembic migration drift gate in CI.

None of that is aspirational — it's shipped and tested. Good. Now the part
that's more useful to share publicly: what's still missing, and why I'm not
rushing to fix all of it.

## Tier 1 — the gaps that actually matter

**1. MCP tool-level OAuth 2.1 + token exchange (RFC 8693)**
Right now the HTTP transport uses one static bearer token for every tool call.
Real 2026 guidance is audience-scoped tokens per tool-domain
(`identity:write` vs `access:grant`) with token exchange instead of forwarding
a client token downstream. This is the deepest, most spec-literal MCP security
work I haven't done yet — and I think it's the single highest-leverage thing
left, because it turns "anyone with the shared key can call any tool" into a
real authorization boundary instead of a convention.

**2. No tool-description integrity check**
The agent re-pulls `tools/list` from the MCP server on every planning step and
feeds whatever comes back straight into the planner prompt. That's the right
architecture for real MCP discovery — but it's also exactly the mechanism
"tool poisoning" attacks exploit: a compromised or MITM'd server mutates a
tool's description between sessions and steers the planner without the human
ever seeing a code diff. I have zero detection for that today. Hash-pinning
discovered tool descriptions (or running something like Invariant Labs'
`mcp-scan`) is the fix, and I haven't built it.

**3. Audit log is a normal, mutable DB table**
Anyone with DB-admin access can edit `AuditLog` rows after the fact. SOC2-grade
expectation is hash-chained or WORM-stored entries, or streaming to an
external SIEM. This is a "looks fine in a demo, fails an actual audit"
category of gap.

**4. No SBOM — correction, this one was wrong**
I flagged this as a gap without actually re-checking `release.yml` first.
It's already there: every tagged release generates an SPDX SBOM
(`anchore/sbom-action`), keylessly signs the image with cosign via Sigstore,
attaches a signed SBOM attestation and an SLSA provenance predicate (both
keyed to the image digest, not the mutable tag), and publishes the SBOM as
a downloadable release asset. Leaving this in the post instead of quietly
deleting it — the honest version of "build in public" includes correcting
your own research when it's wrong, not just listing wins.

**5. No adversarial fuzzing of the guardrails I claim to have**
I have one pinned prompt-injection-refusal test case. NIST's 2026 red-team
research found novel attack techniques hit an 81% task-hijack success rate
against naive agents, vs. 11% for known baseline attacks — meaning "I have a
test for the attack I already thought of" is a much weaker claim than it
sounds. A fuzzing harness (garak / PyRIT style) generating novel injection
variants in CI is the honest next step, not just adding more fixed cases.

**6. Backup/DR shares blast radius with the agent's own credentials**
This agent has write access to identity/access systems through approved tool
calls. There's a real 2026 incident pattern of an agent, using valid approved
credentials, deleting production data *and* its backups because the backup
plane trusted the same credentials as the app plane. I don't yet have a
documented, credential-isolated backup strategy.

**7. Cost governance stops at the per-ticket level**
`MAX_TOKENS_PER_TICKET` exists and is checkpoint-safe across HITL resumes —
genuinely solid. But there's no org-level daily/monthly spend cap or anomaly
alert on top of it, and industry data says 96% of enterprises blow past their
AI cost projections. Per-ticket budgeting alone won't catch a slow bleed.

## Tier 2 — real, but conditional on the actual goal

- **Multi-tenancy** (tenant_id everywhere, bulkhead isolation, per-tenant rate
  limits) — only worth building if the actual plan is selling to multiple
  organizations. Retrofitting it into a single-org deployment would be pure
  waste.
- **SCIM-based identity sync** replacing the mock employee table — the
  architecturally "correct" pattern (HR → IdP → AD, cache never
  system-of-record), deliberately scoped down because a solo project has no
  real HR tenant to sync from.
- **Load/chaos testing** — I have circuit-breaker *unit* tests, not an actual
  load test or a chaos game day against the live stack.
- **Progressive delivery** (canary/blue-green via Argo Rollouts or Flagger) on
  top of the Helm chart that already exists.
- **Formal SLOs/error budgets** tied to the alert rules I already ship.

## What I'm deliberately NOT chasing (and why that's the actual signal)

- **Temporal.** It's real, well-funded engineering, and completely
  disproportionate to this system's actual failure surface (an LLM call and a
  handful of mock tool calls). LangGraph's `RetryPolicy` + a Postgres
  checkpointer covers the risk that actually exists here. Temporal earns its
  keep once there are multi-day human-in-the-loop steps or genuinely
  distributed workers — not before.
- **An external MCP gateway product** (IBM's mcp-context-forge is still
  alpha/unsupported as of this year). My in-process `mount()`-based
  composition gets the same "multi-server, routed" story without operating a
  second piece of infrastructure I'd have to explain away in an interview.
- **Full multi-tenancy**, absent an actual multi-org sales motion.

Knowing what *not* to build is as much a part of "enterprise-grade judgment"
as building the right things — a gap list with no boundaries just reads as a
todo list, not an architecture decision.

## Honest self-correction on "memory"

Worth calling out on its own: I was initially tempted to claim this project
has "agent memory" because the LangGraph checkpointer persists state across
the interrupt/resume boundary and across app restarts. That's real, and it's
the *harder* half of memory to get right. But it is not semantic recall —
nothing here learns across tickets, there's no vector store, no cross-session
memory. Conflating durable-execution state with long-term memory is exactly
the kind of overclaim that erodes credibility with anyone who actually knows
the distinction. So: state persistence, yes. Memory, not yet.
